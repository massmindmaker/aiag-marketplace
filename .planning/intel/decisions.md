## D#2: API Gateway — Hono на Bun

🟡 In progress (Phase 4 active)

Gateway — Hono на Bun runtime (TypeScript-monorepo с frontend).

**Why:**
- TS-monorepo не разваливается, provider adapters шарятся с Next.js
- Bun держит 20–60k RPS на ядро (vs Node)
- Миграция с Node минимальна (Hono тот же)
- Под 50k+ RPS достаточно 2–4 инстансов
- Критичные горячие участки можно переписать на Go выборочно

**Trigger для Go:** оборот > $50k/мес или горячие места не справляются.

## D#9: Routing modes — `mode` параметр в API

🟢 LOCKED 2026-04-18

API принимает `mode: auto / fastest / cheapest / balanced / ru-only`.

| Mode | Формула | Для кого |
|---|---|---|
| `auto` (default) | balanced + override по модели | массовый кейс |
| `fastest` | min(latency_p50) | real-time UX |
| `cheapest` | min(price × markup) | batch / дешёвые задачи |
| `balanced` | 0.5×price + 0.3×latency + 0.2×uptime | по-умолчанию для dev |
| `ru-only` | только RU-residency upstream | enterprise / 152-ФЗ |

В response header: `X-AIAG-Upstream: <name>`.

**Дифференциатор:** ни один конкурент в РФ не даёт routing modes как явный API-параметр.

## D#12: Drop Supabase for MVP

🟢 LOCKED 2026-04-24

**Замена:** Timeweb managed PostgreSQL 16 + NextAuth v5 + Timeweb Cloud S3.

| Было (Supabase) | Стало |
|---|---|
| Supabase Postgres | Timeweb managed PostgreSQL 16 |
| Supabase Auth (GoTrue) | NextAuth v5 (already in repo) |
| Supabase Storage | Timeweb Cloud S3 via @aws-sdk/client-s3 |
| Supabase Realtime | SSE из Hono Gateway |
| Supabase Studio | pgAdmin / DBeaver / drizzle-kit studio |
| Supabase Edge Functions | Hono routes / Next.js handlers |

**Why:**
- Self-hosted Supabase = ~25 контейнеров, нужен 8GB/4vCPU VPS
- Supabase Cloud = данные в AWS → ломает 152-ФЗ
- Realtime в MVP избыточен (SSE покрывает стриминг LLM)

**Economy:** -1300₽/мес инфра, в 5 раз проще ops.

**Trigger возврата:** реальная потребность в Realtime / RLS-RBAC / Edge Functions (≥1000 активных участников live leaderboard, multi-tenant с row-level isolation, edge compute).

Полный разбор: `brain/Knowledge/19-pivot-no-supabase.md`.

## D#13: Drop Docker entirely

🟢 LOCKED 2026-04-24

Все компоненты — нативно через apt + systemd. Postgres, Redis, Nginx, Node — пакеты дистрибутива. Приложения — pm2 под systemd.

**Why:**
- Docker daemon + containerd сами по себе ~200-400 MB до запуска первого контейнера
- На 2GB VPS это критичный overhead для Postgres + 3 Node процессов
- На single-VPS изоляция контейнеров не даёт практической выгоды
- Нативные systemd-units + journalctl дают прямую диагностику

**Trade-off:** upgrades — ручные через apt + тест. Dev ≠ prod (используем SSH-туннель к prod DB для dev).

**Eval sandbox exception:** для Plan 07 — `systemd-run` + `unshare` + `ulimit`. В Phase 2 — nsjail или Docker selectively под sandbox.

**Trigger возврата Docker:** горизонтальное масштабирование на ≥3 VPS (k3s) или hard requirement изоляции для eval sandbox.

## D#14: Drop Dokploy → pm2 + nginx + GH Actions

🟢 LOCKED 2026-04-24

**Replace:**
- **Process supervision:** pm2 (+ systemd supervising pm2 daemon)
- **Reverse proxy + SSL:** Nginx + certbot (Let's Encrypt)
- **Deploy pipeline:** GitHub Actions → SSH → rsync в `releases/<sha>/` → atomic symlink flip → `pm2 reload`
- **Rollback:** symlink на предыдущий release + `pm2 reload`

Capistrano-style:
```
/srv/aiag/<app>/
  releases/<sha>/
  current -> releases/<sha>
shared/.env.production
```

**Why:**
- Dokploy = Traefik + docker-compose + UI + webhook runner. Всё зависит от Docker.
- После D#13 Docker удалён → Dokploy не применим.

**Trade-off:** secret management руками через `/srv/aiag/shared/.env.production` chmod 600. Setup `DEPLOY_SSH_KEY` в repo secrets.

**Trigger возврата PaaS:** команда >3 чел и потребность в self-service UI для deploys / env vars / logs без SSH.

См. `brain/Knowledge/20-bare-metal-stack.md` для полного описания.
